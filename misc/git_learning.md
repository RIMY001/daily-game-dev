## 2026-06-09

### 让千问总结git命令

Git 的核心工作流程可以概括为：**修改文件 → 添加到暂存区 → 提交到本地仓库 → 推送到远程仓库**。将常用命令按日常开发场景进行分类总结：

新手入门：本地项目首次上传到 GitHub
当有一个全新的本地项目，想要第一次推送到远程仓库时，标准流程如下：
git init：在当前文件夹初始化 Git 仓库（生成隐藏的 .git 目录）。
git add .：将当前目录下所有新增、修改的文件添加到暂存区。
git commit -m "初始化项目"：将暂存区的内容正式提交到本地仓库，并写上清晰的提交说明。
git branch -M main：将当前主分支重命名为 main（GitHub 默认主分支名）。
git remote add origin <你的仓库地址>：将本地仓库与 GitHub 上的远程仓库关联起来。
git push -u origin main：首次推送代码到远程仓库。-u 参数会建立本地和远程分支的追踪关系，以后只需输入 git push 即可。

日常开发：高频核心命令
在平时的代码编写中，你只需要反复使用以下几个命令：
git status：查看当前仓库状态。这是最重要的“安全锁”，能告诉你哪些文件被修改了、哪些在暂存区、当前在哪个分支。
git add <文件名> 或 git add .：精准添加单个文件或批量添加所有改动到暂存区。
git commit -m "feat: 新增登录功能"：将改动保存到本地版本库。建议结合你笔记中提到的规范，使用 feat:、fix: 等前缀。
git push：将本地提交推送到远程仓库。

团队协作：分支管理
在团队开发中，通常不建议直接在主分支（main）上修改代码，而是采用分支工作流：
git pull：每天写代码前，先拉取并合并远程的最新代码，避免冲突。
git checkout -b feature/xxx 或 git switch -c feature/xxx：基于当前分支创建并切换到一个新的功能分支。
git merge feature/xxx：当新功能开发完毕，切回主分支后，将该功能分支合并进来。
git branch -d feature/xxx：删除已经合并的无用分支。

实用技巧：查看历史与临时切换任务
git log --oneline：以单行简洁模式查看提交历史，方便追溯和寻找历史版本的哈希值（commit id）。
git diff：查看工作区文件与暂存区的具体代码差异（新增、删除了哪些行）。
git stash：当你代码写了一半，突然需要去修复紧急 Bug 时，用它把当前未完成的修改“藏起来”。
git stash pop：处理完紧急任务切回原分支后，把之前“藏起来”的代码恢复出来继续开发。

核心概念提示：
理解 Git 的关键在于记住它的“三个区域”：工作区（正在编辑的草稿纸） → git add → 暂存区（打包台，准备提交的内容） → git commit → 版本库（正式归档的历史记录）。

### ai知识手册
**Git 命令语法规范 —— 空格、参数与选项的工程解析**

**关键词**：命令语法、参数解析、选项标识、语义化提交

#### 一、核心原则：Git 命令的基本语法结构
Git 命令遵循 **Unix 命令行通用范式**，其基础语法结构可概括为：
```命令 [选项] [参数]```
其中：
- **命令（Command）**：动词，指明要执行的操作，如 add、commit、push。
- **选项（Option / Flag）**：修饰词，用于调整命令的具体行为，以 - 或 -- 为前缀。
- **参数（Argument）**：宾语，指定命令操作的对象，如文件路径、分支名、提交信息等。

**核心约束**：上述三者之间**必须由空格分隔**。空格是 Git 解析器识别语义单元的唯一依据。

#### 二、空格（Space）：命令解析的“边界符”
在 Git 及绝大多数命令行工具中，空格（）被定义为 **分隔符（Delimiter**）。其功能是将输入字符串切分为独立的标记（Token），供系统逐一解析。

2.1 正确示例
```git add .```
解析结果：
标记 1：git → 主命令
标记 2：add → 子命令
标记 3：. → 参数（代表当前目录）

2.2 常见错误
```git add.```
解析结果：
系统将 add. 视为单一的未识别命令，因而报错。

**工程规范**：任何语义单元之间都必须通过空格显式分隔，不得省略。

#### 三、短横线（- / --）：选项的“标识符”
短横线是选项的**强制性前缀**，用于区分“操作方式”与“操作对象”。这是 Git 及其他 CLI 工具的通用约定。

3.1 单短横线（-）
用于**缩写形式的选项**，通常为单个字母，便于快速输入。
```
选项	完整写法		功能说明
-m	--message		指定提交信息
-b	--branch		创建并切换到新分支
-u	--set-upstream	关联远程分支（首次推送）
-f	--force		强制推送（覆写远程）
```
示例：
```git commit -m "feat: 添加排序算法"```
3.2 双短横线（--）
用于完整单词形式的选项，增强可读性，常用于配置类或复杂场景。
```
选项	功能说明
--amend	修改最近一次提交内容
--global	设置全局配置
--oneline	日志单行显示
--graph	图形化显示分支历史
```
示例：
```git commit --amend -m "修正提交信息"```
3.3 双短横线作为“分隔符”（--）
当参数与选项名称可能发生歧义时（例如文件名恰好为 -m），使用 -- 强制终止选项解析，后续内容全部视为参数。

示例：
```git add -- -m```
含义：将名为 -m 的文件添加到暂存区，而非误认为 -m 是选项。

#### 四、参数（Argument）：操作的对象
参数是指定 “对谁操作” 的对象，不以横线为前缀，用于补充命令的具体内容。

4.1 常见参数类型与示例
```
参数类别		示例						说明
当前目录		.						代表当前文件夹及所有子内容
父目录		..						代表上一级文件夹
文件路径		main.cpp					指定单个文件
文件夹路径		src/						指定整个文件夹
分支名		main、develop				指定操作的分支
远程仓库别名	origin					默认远程仓库标识
URL 地址		https://github.com/...			指定远程仓库位置
提交哈希值		a1b2c3d					指向某次具体的提交
HEAD		HEAD					当前分支的最新提交指针
提交信息		"fix: 修复空指针"				与 -m 选项配合使用
```
示例：
```
git add cpp/07_integrated_projects/
git push origin main
git checkout a1b2c3d
```
#### 五、选项与参数的对比速查表
```
特征维度	**选项（Option）**		**参数（Argument）**
语义角色	修饰命令的“方式/开关”	指定操作的“对象/数据”
是否加 -	**必须加**（- 或 --）		**绝对不加**
顺序		通常位于参数之前		通常位于选项之后
功能		改变命令的执行逻辑		提供操作的目标信息
示例		-m、--amend、-f		.、main、day41.cpp
```
#### 六、工程实践建议
1. **空格必须严格**：每次输入命令时，确保每个语义单元之间有且仅有一个空格。

2. **选项前置**：选项（以 - 开头）通常写在参数之前，如 git commit -m "信息"。

3. **用 -- 消除歧义**：当文件名与选项名称冲突时，使用 -- 明确分隔。

4. **善用 `git status `检查**：在执行提交前，先查看当前状态，确认改动已正确暂存。
